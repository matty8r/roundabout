# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Roundabout is a macOS menu-bar app (Swift Package, not an Xcode project) that tracks the user's actual working "contexts" — which Terminal tab/process, which Safari tab, which app — and provides an Option-Tab / Option-Shift-Tab switcher (styled with real Liquid Glass, `NSGlassEffectView`) that jumps straight back into the selected context, not just the app. It's packaged as a login item (`SMAppService`, toggled from the menu bar), so day-to-day it runs continuously in the background rather than being launched by hand each session. (Originally built and shipped internally under the name "Breadcrumbs" — some legacy identifiers, like the Keychain fallback in `APIKeyStore`, still reference that name on purpose; see below.)

## Commands

Build:
```
swift build
```

Package as a `.app` bundle (**required** before testing anything that needs Accessibility or Automation permissions — macOS will not reliably grant those to a raw `.build/debug/Roundabout` executable, only to a proper bundle):
```
./scripts/build_app.sh [debug|release]   # defaults to debug
```

Dev-loop run (kill any existing instance first; launch from a shell so the process inherits `ANTHROPIC_API_KEY`, and so stderr is visible live in this terminal in addition to the durable log described below):
```
pkill -f "Roundabout" 2>/dev/null
./scripts/build_app.sh debug
source .env   # loads ANTHROPIC_API_KEY (gitignored, not in shell rc)
nohup ./Roundabout.app/Contents/MacOS/Roundabout > /tmp/roundabout.log 2>&1 &
disown
```

Installing the "real" copy (the one the login item / Launch-at-Login toggle actually runs) — after a rebuild, replace the copy in `/Applications` so the running instance reflects the latest code:
```
pkill -f "Roundabout" 2>/dev/null
rm -rf /Applications/Roundabout.app
cp -R Roundabout.app /Applications/Roundabout.app
```
Then either relaunch it manually (`/Applications/Roundabout.app/Contents/MacOS/Roundabout &`, with `.env` sourced first) or just let the login item start it on next login.

There is no test target. Verification is done live:
- **Durable log** at `~/Library/Application Support/Roundabout/roundabout.log` — written via `Log.write()` (`Logger.swift`), which duplicates every message to both stderr and this file. This is the log to check for a login-item launch specifically: the OS starts that process directly with no shell attached, so `/tmp/roundabout.log`-style redirection never happens for it, and stderr alone would be invisible. The dev-loop run above still also gets a live stderr stream via `/tmp/roundabout.log`.
- **SQLite store**, queried directly, e.g.:
```
sqlite3 ~/Library/Application\ Support/Roundabout/roundabout.sqlite3 "SELECT source, app, cwd, url, is_active_now FROM snapshots ORDER BY id DESC LIMIT 20;"
```

Whenever the SQLite schema changes (a new column added to `SnapshotStore`), delete the on-disk database before relaunching, or old rows will read back with the new fields defaulted/null and produce confusing results:
```
rm -f ~/Library/Application\ Support/Roundabout/roundabout.sqlite3
```

## Architecture

Data flow: **Collectors → SnapshotStore (SQLite) → ContextClusterer → a `Summarizer` backend (async, gated) → StatusItemController (menu) + SwitcherWindowController (Option-Tab overlay) → ContextActivator (jump back)**. `AppDelegate` owns all of it: a 15s poll `Timer` driving `tick()`, an `NSWorkspace.didActivateApplicationNotification` observer for instant (non-polled) app-switch tracking, the store, the hotkey manager, the switcher controller, and the summary cache/in-flight tracking.

**Collectors** (`Sources/Roundabout/Collectors/`) each produce `[Snapshot]` from one signal source:
- `FrontmostAppCollector` — `NSWorkspace.frontmostApplication`; inherently "active" whenever it fires, since it only ever reports the current frontmost app. Filters to `activationPolicy == .regular` — without this, system/background agents that can briefly become "frontmost" (`loginwindow`, `UserNotificationCenter`, and notably `universalAccessAuthWarn`, which pops up when Roundabout itself requests Accessibility permission) show up as bogus contexts. Also excludes Terminal and Safari by name (`isCoveredByDedicatedCollector`) — they already have dedicated collectors producing tab-specific contexts, so without this exclusion a generic "app:Terminal"/"app:Safari" fallback context would appear *alongside* the real one, competing for a switcher slot despite being strictly less useful (no cwd/tty to jump to). The same filter is applied again in `AppDelegate.appActivated(_:)`, the event-driven counterpart to this collector.
- `TerminalCollector` — AppleScript enumerates Terminal windows/tabs; resolves each tty's cwd and foreground process via `ps`/`lsof`.
- `SafariCollector` — AppleScript enumerates Safari windows/tabs (URL, title); gated on Safari already running so polling doesn't launch it. Also exposes `fetchPageText(forURL:)`, used by the summarizer (see below).

Both AppleScript collectors share three non-obvious gotchas, already the cause of separate bugs — don't reintroduce them:
- Inside `tell application "Terminal"` / `"Safari"`, the bare `tab` keyword is shadowed by the app's own "tab" (window/browser tab) scripting noun and silently stringifies to the literal text `"tab"` instead of a tab character. Use `(ASCII character 9)` for delimiters.
- AppleScript's `is` operator for object identity does not reliably compare two separately-fetched references to the same tab/window (e.g. `t is (selected tab of w)` can evaluate false even when `t` genuinely is the selected tab). Compare by a concrete value instead — tty string, tab `index`, window `id`.
- A missing/denied Automation permission makes `osascript` fail, not hang — but each collector's `fetchTabs()` captures stderr specifically (via a `runProcessCapturingStderr` helper distinct from the plain stderr-discarding `runProcess` used for `ps`/`lsof` in `TerminalCollector`) and logs it, rather than just returning `[]` and looking identical to "no tabs open." This was a real bug found via a from-scratch clean-install test: Automation entries for Terminal/Safari can get pre-created in a *disabled* state with no prompt ever shown (see `NSAppleEventsUsageDescription` below) — without this logging, that failure mode was completely silent and undiagnosable.

**`Snapshot`** (`Models/Snapshot.swift`) is the raw per-poll/per-event record. The important field is `isActiveNow`: true only when the snapshot is the system's frontmost app's frontmost window's selected tab *at that moment*. This is what drives switcher ordering — treating "was observed to exist" as "recently used" (the original bug) means every open tab gets bumped to "now" every poll and whichever collector happens to run last in `tick()` wins the recency sort regardless of actual attention. `Context.domain` (same file) is a computed property, not a stored/collected field — it parses the bare host (no "www.") straight out of `url` for display, so a Safari row has a second line immediately, without waiting on an async summary.

**`SnapshotStore`** (`Storage/SnapshotStore.swift`) is an append-only SQLite log at `~/Library/Application Support/Roundabout/roundabout.sqlite3`, using the raw C SQLite3 API (no ORM). Every field on `Snapshot` must be threaded through `CREATE TABLE`, `INSERT`, and `SELECT` by hand — a field added to the Swift struct but not the SQL layer silently reads back as nil/default with no compile error (has already happened twice: `processName`/`isActiveNow`, then `url`/`is_active_now`).

**`ContextClusterer`** (`Clustering/ContextClusterer.swift`) groups snapshots into `Context`s by key: `cwd::processName` for terminal snapshots (so an idle shell and an active `claude` session in the same directory become distinct contexts, not one merged/ambiguous one), the page `url` for Safari tabs, `app:<name>` as the fallback. The key derivation is exposed as a standalone `ContextClusterer.key(for:)` so `AppDelegate` can reuse the exact same logic to check "is this specific context still open" (see pruning, below) without duplicating the branching. `lastSeen` only advances on `isActiveNow == true` snapshots. It also tracks a "representative" tty/url per key (preferring the currently-selected tab) for `ContextActivator` to jump to.

**Pruning** happens in `AppDelegate.refreshAndRender()`, not in the clusterer: after each full `tick()`, the freshly-collected snapshots are cached as `lastFreshSnapshots`, and every render (whether triggered by a poll tick, an app-activation event, or an async summary arriving) filters `contexts` down to only those whose key is in `lastFreshSnapshots` — except `app:`-prefixed fallback contexts, which are checked against `NSWorkspace.runningApplications` instead, since no collector enumerates every running app on every tick. Pruning against a cache (rather than only pruning on the render that did the fresh collection) matters: a render that skipped pruning could silently "revive" a just-closed tab/app until the next full poll happened to land.

**`Summarizer`** (`Summarizer/Summarizer.swift`) is the protocol both backends conform to — `summarize(cwd:label:) async -> SummarizerResult?` and `summarizeWebPage(title:excerpt:) async -> SummarizerResult?`, both returning nil (caller keeps the cheap default label) rather than throwing, since a summary is always an optional enhancement. `SummarizerPrompts` in the same file holds the actual prompt text so both backends ask the same question — switching providers shouldn't also silently change what's being asked for. `AppDelegate.refreshAndRender()` decides *when* to call these, not the summarizer itself: terminal contexts only when `processName == "claude"` (an idle shell sharing a cwd with an active session would otherwise inherit that session's misleading summary); every Safari context gets summarized unconditionally now — the original "only when the cheap title collides with another open tab's" gate existed to limit paid API calls, and no longer earns its complexity now that the default backend (on-device) is free. `AppDelegate.activeSummarizer` resolves which backend to use fresh on every call (not cached), by reading `SummarizerPreferenceStore.current` — so flipping the status-bar menu's "Summarization" selection takes effect on the very next pass, no relaunch needed; a request already in flight when the user switches finishes with whichever backend it started with (the summarizer is captured once, right before the `Task` that awaits it, not read again inside the closure). Note the on-device backend can produce a confidently-wrong summary on thin/failed page-text extraction (small local models hallucinate more readily than Haiku) — there's no code-level guard against this, just switch that context to Anthropic if it bothers you.

- **`FoundationModelsSummarizer`** (`Summarizer/FoundationModelsSummarizer.swift`) — **the default.** On-device via Apple's `FoundationModels` framework (`SystemLanguageModel` / `LanguageModelSession`), macOS 26+ only. Free, private (nothing leaves the device), no API key. `@Generable`/`@Guide` macros define a `GeneratedSummary` struct (`name`, `summary`) passed to `session.respond(to:generating:)` for structured output — note the `@Generable` type must be at least `fileprivate`, not `private`: the macro's generated `extension GeneratedSummary: Generable` sits in the same file but outside the enclosing type, and `private` would make the type inaccessible to its own generated extension (a `private` protection level, not a `Generable`-specific restriction — this is a general Swift access-control interaction with macro-generated peer/extension code). `isAvailable` / `unavailableReason` wrap `SystemLanguageModel.default.availability`, whose `.unavailable` case carries `.deviceNotEligible` / `.appleIntelligenceNotEnabled` / `.modelNotReady` — surfaced via the "On-Device" menu item's tooltip so an unavailable state is diagnosable rather than a silent no-op. Deliberately has **no fallback to Anthropic** when unavailable: a user who picked on-device specifically for privacy shouldn't have requests silently start leaving the device just because Apple Intelligence isn't enabled today — that's an explicit provider switch, not something this does on its own.
- **`ClaudeSummarizer`** (`Summarizer/ClaudeSummarizer.swift`) — **opt-in, for power users.** Makes raw HTTPS calls to the Anthropic Messages API (no official Swift SDK exists) using structured output (`output_config.format` / `json_schema`) to get back `{name, summary}`. Uses `claude-haiku-4-5` deliberately, for cost, since it fires on live/changing contexts. Resolves the API key via `APIKeyStore` (Keychain) first, falling back to `ANTHROPIC_API_KEY` in the process environment — the Keychain path is what makes summarization work when Roundabout is running as a login item, since that launch path is started directly by `launchd` with no shell involved and never sees a shell-exported env var; the env-var fallback exists purely so the dev-loop run (`source .env` before launching) still works without also requiring a Keychain entry. The key is set/cleared via "Set Anthropic API Key…" / "Clear Anthropic API Key" in the status-bar menu (`StatusItemController`), which stores it with `kSecAttrAccessibleAfterFirstUnlock` so it's readable even before an interactive unlock. `APIKeyStore` also reads a `legacyService` Keychain identifier (`com.msilas.breadcrumbs`, from before the app was renamed) as a fallback if the current-name entry is empty, purely so renaming the app didn't silently orphan a key someone had already saved.
  - `summarize(cwd:label:)` — reads the matching Claude Code session transcript from `~/.claude/projects/-<sanitized-cwd>/` via `TranscriptReader`.
  - `summarizeWebPage(title:excerpt:)` — takes page text fetched by `SafariCollector.fetchPageText` (`do JavaScript` in the matched tab — requires the user to enable Safari's Develop menu → "Allow JavaScript from Apple Events"; without it, this fails loudly to the log rather than silently no-opping).

**`SummarizerPreferenceStore`** (`SummarizerPreferenceStore.swift`) persists the chosen provider (`.onDevice` / `.anthropic`) in plain `UserDefaults` — a preference, not a secret, so no Keychain involvement. Defaults to `.onDevice` when unset.

**`ContextActivator`** (`Collectors/ContextActivator.swift`) is "jump to this context": AppleScript to select the exact Terminal tab (by tty) or Safari tab (by URL) and bring its window forward, or `NSRunningApplication.activate()` for app-only fallback contexts. `AppDelegate.markActivated(_:)` runs immediately after every `ContextActivator.activate(_:)` call from the switcher — it inserts a synthetic `isActiveNow` snapshot for the just-activated context and re-renders, so that context's `lastSeen` is current *before* the next hotkey press rather than waiting for the next 15s poll (or, for Terminal/Safari specifically, waiting on nothing at all, since `appActivated` doesn't track them — see `FrontmostAppCollector`). Without this, a quick single Option-Tab tap — meant to mirror Cmd-Tab's "jump straight back" gesture — wouldn't reliably ping-pong between the two most recent contexts on repeated taps, since `latestContexts` wouldn't yet reflect that the context you just jumped to is now the current one.

**`HotkeyManager`** (`Hotkey/HotkeyManager.swift`) is a `CGEventTap`-based global listener for Option-Tab (cycle) and Option-Shift-Tab (cycle backward) — Cmd-Tab itself can't be intercepted since the system owns it. Requires Accessibility trust (`AXIsProcessTrustedWithOptions`); the tap silently produces nothing without it. `AppDelegate` logs both the trust check result and whether the tap was created successfully, unconditionally, on every launch (see `Log.write` below) — that's specifically so a launch that succeeds silently isn't indistinguishable from one that never logged anything. If trust is missing at launch, `AppDelegate.waitForAccessibilityPermission()` polls `HotkeyManager.isAccessibilityTrusted()` (a plain `AXIsProcessTrusted()` check, no prompt — repeatedly calling the prompting variant would re-trigger the system dialog/Dock-bounce on every poll) every 2s and starts the hotkey the moment it goes true, so granting the permission in System Settings while Roundabout is already running is enough on its own — no quit-and-relaunch cycle needed. This only eliminates the "relaunch" step of the old workflow, not the "go grant it" step itself — ad-hoc signing can still mean System Settings' toggle doesn't visibly "take" until the existing Accessibility entry is removed and re-added rather than just switched on (see Packaging below).

**`Log`** (`Logger.swift`) writes every diagnostic message to both stderr and a persistent file at `~/Library/Application Support/Roundabout/roundabout.log`. This exists because a login-item launch is started directly by the OS with no shell to redirect stderr from — `fputs(..., stderr)` alone is invisible for exactly that launch path. Use `Log.write(...)`, not raw `fputs`, for any new diagnostics.

**`LoginItemManager`** (`LoginItemManager.swift`) wraps `SMAppService.mainApp` to register/unregister Roundabout as a login item — the modern replacement for a `~/Library/LaunchAgents` plist. Toggled via "Launch at Login" in the status-bar menu (`StatusItemController`), which re-reads `LoginItemManager.isEnabled` after toggling rather than assuming success, since the OS can come back with `.requiresApproval` (in which case System Settings' Login Items pane is opened automatically). Note: `SMAppService.mainApp` is keyed to the app's bundle identifier (`com.msilas.roundabout`) — a login item registered under the old `com.msilas.breadcrumbs` identity is orphaned by the rename and won't show up as "enabled" here; toggling "Launch at Login" off/on once re-registers it under the new identity.

**`SwitcherWindowController` / `ContextRowView`** (`Switcher/`) render the Option-Tab overlay: a borderless `NSPanel` containing a single top-level `NSGlassEffectView` (real Liquid Glass — macOS 26+ only, which is why `Package.swift` targets `.macOS(.v26)`). Rows are NOT nested glass views (unsupported z-ordering per Apple's docs) — highlighting is a plain layered `CALayer`. Rows size themselves (summaries wrap instead of truncating), so panel height comes from `stack.fittingSize`, clamped to 85% of screen height. Each row's text stack is up to three lines: title, then `context.domain` (browser contexts only, hidden otherwise), then `context.summary` once one exists — domain is always there immediately since it's cheap/synchronous, summary arrives async. `ContextRowView` is also reused directly by `StatusItemController` for the status-bar dropdown's context rows — assigning it as an `NSMenuItem.view` gets both wrapping, non-grayed text (a custom view draws its own colors regardless of the item's `isEnabled` state, bypassing `NSMenuItem`'s default single-line auto-grayed title rendering) and click-to-jump (`ContextRowView.onSelect`, wired to `AppDelegate` the same way the switcher's own selection is: `ContextActivator.activate` + `markActivated`). Two non-obvious things this needed: (1) `isEnabled = true` on the item — with a custom view, `isEnabled` no longer controls appearance (the view owns that), only whether the item can hover-highlight/receive clicks at all; (2) the view's `frame` must be forced via `layoutSubtreeIfNeeded()` + reading `fittingSize` *before* assigning it as `item.view` — `NSMenu` reads a custom view's frame synchronously at assignment time to reserve that item's hit-test region, it does not wait for the Auto-Layout pass that happens later when the menu actually draws, so skipping this leaves every row's *click* region computed from a stale zero-sized rect even though rendering (which happens after layout resolves) looks completely correct — this is what caused an earlier bug where the dropdown looked right but no item, including ones below the context rows, was clickable.

**Choosing a summarization provider**: the status-bar menu's "Summarization" submenu (built by `StatusItemController.makeSummarizationMenu()`) is a radio-button-style pair of checkable items — "On-Device (Apple Intelligence)" and "Anthropic Claude" — reflecting and writing `SummarizerPreferenceStore.current`. Picking either calls `AppDelegate`'s `onSummarizerPreferenceChanged` hook, which clears `summaryCache` and re-renders immediately, so switching providers is felt right away rather than waiting up to 5 minutes for the old cache entries to expire.

**App icon**: `Resources/AppIcon.svg` is the source-of-truth logo — a UK "roundabout circulation" mandatory road sign (blue disc, three white curved arrows in rotational symmetry indicating clockwise flow) mounted on a grey post, standing against a full-bleed sky-blue-to-grass-green gradient, so it reads as a literal roadside sign rather than a flat pictogram floating on transparency. The sign itself is drawn once at its original size/coordinates (centered at (50,50), r=46) and repositioned via a single `<g transform="translate(cx,cy) scale(s) translate(-50,-50)">` — translate original center to origin, scale, translate to the new center — rather than hand-recomputing every arrow/arrowhead coordinate for a new size and position; the post fills the vertical space that move frees up at the bottom. `scripts/build_app.sh` rasterizes this SVG at each standard icon size via `sips` and bundles the result into `Roundabout.icns` via `iconutil`, referenced by `Info.plist`'s `CFBundleIconFile` — this is what Finder/System Settings' Login Items list/the eventual About panel show. **Menu bar glyph** is a separate, simpler asset: `Resources/MenuBarIcon.svg` is just the three arrows (no disc, no gradient, no post) cropped from the *original* centered/pre-repositioning coordinates, loaded directly at runtime (`NSImage(contentsOf:)`) as a *template* image so AppKit recolors it to match the system's other menu bar items — a filled disc/gradient wouldn't read well as a small monochrome status item, and a template image ignores fill color entirely (only the alpha channel matters), so there'd be no point giving it color anyway.

**Packaging**: this is a Swift Package executable, not an Xcode project, so there's no `.app` bundle by default — `scripts/build_app.sh` assembles one (`Resources/Info.plist`, the generated `.icns`, ad-hoc `codesign`) after every build. Permissions (Accessibility, Safari Automation) must be granted to that bundle, not the raw binary — and re-granted if the bundle is ever rebuilt at a materially different path, since ad-hoc signing has no stable Team ID for TCC to key off of as reliably as a real Developer ID signature would. Renaming the app (as happened going from Breadcrumbs to Roundabout) has the same effect as a rebuild-at-a-different-path for TCC purposes — expect to re-grant Accessibility/Automation permissions once after a rename, in addition to the login-item and Keychain identity notes above. `Info.plist` declares `NSAppleEventsUsageDescription`, required for macOS to show the Automation authorization *prompt* at all when Terminal/Safari's AppleScript is first invoked — confirmed missing via a from-scratch clean-install test where Terminal/Safari Automation entries showed up in System Settings pre-created but disabled, with no prompt ever having appeared; unlike Accessibility, an existing (even disabled) Automation record blocks any future prompt for that pairing, so this key only helps on a genuinely first-ever grant, not on recovering from a stuck denied one.

**Swift language mode**: `Package.swift` pins `swiftLanguageModes: [.v5]` despite `swift-tools-version:6.2` — the newer tools version is needed for the `.v26` platform case, but Swift 6's default strict-concurrency checking would otherwise force `@MainActor` annotations through this AppKit-heavy, effectively single-threaded codebase.

## Future possibilities (not yet implemented)

- **Shipping a bundled lightweight local model (e.g. Gemma)** as a fallback for Macs that can't run Apple Intelligence at all (`FoundationModelsSummarizer.isAvailable == false` with no Anthropic key configured either) — currently those users just get the cheap default label with no generated summary.
- A third/fourth entry in the "Summarization" submenu for other cloud providers, if there's ever demand beyond Anthropic.
- Additional browser support: Chrome/Arc are cheap to add (same AppleScript dictionary shape as Safari — windows/tabs/URL/title). Firefox has no AppleScript scripting dictionary at all and would need a browser extension + native messaging host — a materially bigger project.
- iTerm2 support (richer AppleScript API than Terminal.app) if the user switches terminal emulators.
